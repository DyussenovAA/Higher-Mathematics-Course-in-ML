Ниже — полный список прод-навыков, которые отличают Kaggle-новичка от ML-инженера, готового к реальной задаче.

Без воды.

1️⃣ Постановка задачи (Problem Framing)

Ты должен уметь:

— перевести бизнес-цель в ML-формулировку
— выбрать тип задачи (binary, multi-class, regression)
— определить целевую метрику
— понять, что стоит ошибка

Ключевой навык: не “строить модель”, а “решать проблему”.

2️⃣ Cost / Risk мышление

Ты обязан:

— считать cost matrix
— понимать FP vs FN
— выбирать threshold осознанно
— считать expected value

Это критично для fintech.

3️⃣ Baseline мышление

Ты должен:

— строить простую модель первой
— понимать, что baseline — точка сравнения
— уметь объяснить, почему новая модель лучше

Без baseline ты не понимаешь, есть ли прогресс.

4️⃣ Data Understanding

Ты обязан:

— анализировать распределения
— находить дисбаланс
— выявлять утечки данных (data leakage)
— понимать корреляции
— проверять дубликаты

Очень часто junior валятся именно здесь.

5️⃣ Feature Engineering

Ты должен уметь:

— создавать агрегаты
— делать временные признаки
— кодировать категории
— масштабировать признаки
— удалять нерелевантные

Фичи часто важнее модели.

6️⃣ Валидация

Обязан знать:

— train/test split
— кросс-валидацию
— stratified split
— time-series split (если временные данные)

Неправильная валидация = фейковый результат.

7️⃣ Метрики

Ты должен:

— выбирать метрику под задачу
— понимать ограничения accuracy
— использовать precision/recall/F1
— понимать ROC-AUC
— анализировать confusion matrix

8️⃣ Сравнение моделей

Ты обязан:

— тестировать несколько моделей
— анализировать bias/variance
— понимать, где переобучение
— сравнивать по стабильности

9️⃣ Анализ ошибок

Это уровень выше среднего.

Ты должен:

— смотреть на false positives
— смотреть на false negatives
— искать закономерности ошибок
— понимать, почему модель ошибается

🔟 Интерпретируемость

В fintech это важно.

Ты должен:

— понимать feature importance
— уметь объяснить вклад признаков
— понимать SHAP на концептуальном уровне

1️⃣1️⃣ Работа с дисбалансом

Ты обязан знать:

— class weights
— oversampling / undersampling
— SMOTE (концептуально)
— threshold tuning

1️⃣2️⃣ Гиперпараметры

Ты должен:

— понимать learning rate
— понимать depth дерева
— понимать регуляризацию
— делать grid search / random search

1️⃣3️⃣ Репликабельность

Прод-мышление включает:

— фиксированный random seed
— воспроизводимость эксперимента
— понятная структура кода
— логирование результатов

1️⃣4️⃣ Минимальный MLOps уровень

Для junior достаточно:

— сохранять модель
— загружать модель
— понимать, как задеплоить API
— понимать проблему drift

1️⃣5️⃣ Документирование решений

Ты должен:

— объяснить, почему выбрал модель
— объяснить, почему выбрал метрику
— показать trade-offs

Итоговый портрет прод-мышления

Ты:

✔ начинаешь с проблемы
✔ считаешь цену ошибки
✔ строишь baseline
✔ сравниваешь модели
✔ анализируешь ошибки
✔ делаешь вывод

LLM-ПРОТОКОЛ ДЛЯ РОСТА
Уровень 0 — Определи тип задачи

Перед тем как обращаться к LLM, классифицируй задачу:

A) Концептуальная (понимание идеи)
B) API/синтаксис
C) Архитектурная
D) Отладка
E) Математика/доказательство
F) Алгоритмическая задача

От типа зависит допустимый способ использования.

Уровень 1 — 30-минутное правило

Перед LLM:

— 30 минут самостоятельной работы
— чтение документации
— попытка сформулировать гипотезу

Если ты не пытался — LLM запрещён.

Иначе ты тренируешь зависимость.

Уровень 2 — Формат правильного запроса

Запрещено:
“Объясни всё с нуля.”

Разрешено:
“Вот моя гипотеза X. Где ошибка?”
“Я думаю, что регуляризация уменьшает variance через ..., правильно ли?”
“Вот код, он падает с ошибкой Y, почему?”

Ты должен приходить с мыслью, не с пустотой.

Уровень 3 — Ограничение помощи
Для математики

LLM можно использовать:
— для уточнения определения
— для проверки логики

Нельзя:
— читать готовые доказательства до своей попытки
— решать задачи через LLM

Иначе ШАД можно забыть.

Для алгоритмов

LLM:
— для ревью
— для обсуждения сложности

Нельзя:
— получать готовое решение
— смотреть оптимальный код

Для ML

LLM:
— обсуждать trade-offs
— проверять reasoning
— анализировать пайплайн

Нельзя:
— генерировать готовое тестовое
— просить “напиши весь pipeline”

Для API / pandas / SQL

LLM — допустим чаще.

Но после ответа:

— открыть документацию
— сверить параметры
— переписать код самому

Уровень 4 — Переписывание

Любой полученный ответ:

Ты обязан:

— закрыть LLM
— воспроизвести решение с нуля
— объяснить его вслух

Если не можешь — значит ты не понял.

Уровень 5 — Проверка глубины

Спроси себя:

“Смогу ли я объяснить это на собеседовании без LLM?”

Если нет — значит ты просто прочитал.

Опасные сигналы зависимости

— ты открываешь LLM сразу
— ты чувствуешь тревогу без него
— ты не читаешь документацию
— ты не можешь повторить решение

Если это происходит — ограничивай использование.

Оптимальное распределение для тебя

Матан / линал → 80% без LLM
Алгоритмы → 70% без LLM
ML-архитектура → 50/50
Backend / стек → 40% без LLM

Почему это важно именно тебе

Ты хочешь:

— ШАД
— ML
— инженерный уровень
— системное мышление

Если LLM станет костылём,
твоя когнитивная мощность не вырастет.

А без роста базы ML и Stabrium не взлетят.
